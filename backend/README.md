# SkillHub Job API

FastAPI backend powering job search, AI-assisted skill matching, and job
extraction for SkillHub.

## Stack
- FastAPI + Uvicorn
- PostgreSQL (SQLAlchemy) for user accounts
- Redis for caching
- Celery for background jobs
- JobSpy for multi-board job scraping (LinkedIn, Indeed, Glassdoor)

## Local setup

```bash
python -m venv venv
source venv/bin/activate  # Windows: venv\Scripts\activate
pip install -r requirements.txt

cp .env.example .env
# Edit .env: at minimum set a real SECRET_KEY and your DATABASE_URL/REDIS_URL
python -c "import secrets; print(secrets.token_hex(32))"  # generate SECRET_KEY

uvicorn app.main:app --reload --port 8001
```

API docs are served at `/docs` (Swagger) and `/redoc`.

## Environment variables
See `.env.example` for the full list. Notes:
- `SECRET_KEY` — the app **refuses to start** if `DEBUG=False` and this is
  left at the placeholder value. Always generate a real one before deploying.
- `DEBUG` — must be `False` (or `production`/`release`) in any deployed
  environment.
- `OPENAI_API_KEY`, `TWITTER_*`, `LINKEDIN_*` — optional; features that use
  them degrade gracefully when unset.

## Authentication
`POST /api/auth/login` is real, DB-backed auth (bcrypt-hashed passwords,
JWT access tokens). There's no separate signup endpoint yet — the first
successful login for a given email creates the account; subsequent logins
verify against the stored password hash. Protected routes (e.g. `/match`)
require an `Authorization: Bearer <token>` header.

## Deployment
- **Docker**: `docker compose up --build` (see `docker-compose.yml`), or
  build `Dockerfile` directly. `.dockerignore` keeps `.env` and other
  local-only files out of the image — never bake real secrets into the
  image; inject them as environment variables at deploy time instead.
- **Render**: `render.yaml` defines a web service + managed Postgres
  database. `SECRET_KEY` is auto-generated by Render; set `REDIS_URL` to
  your Redis instance's connection string.

## Scaling & capacity

What the code now does for you:
- **Async everywhere on the hot paths** — DB access uses async SQLAlchemy +
  asyncpg (not a blocking sync engine), so a slow query doesn't stall other
  requests on the same process.
- **Search results are cached in Redis** (`SEARCH_CACHE_TTL_SECONDS`,
  default 5 min) — repeated/popular queries don't re-scrape LinkedIn/Indeed/
  Glassdoor every time.
- **Scrape concurrency is capped** (`MAX_CONCURRENT_SCRAPES`) so a burst of
  simultaneous searches queues briefly instead of hammering external job
  boards all at once (which is how you get the server's IP rate-limited or
  banned).
- **Rate limiting is enforced** per-IP on search/match/login (`slowapi`),
  protecting both this API and the external sites it scrapes.
- **Runs multi-process** — `Dockerfile` uses `gunicorn` + `WEB_CONCURRENCY`
  uvicorn workers instead of a single process, so it uses all CPU cores in
  a container.

What still requires infrastructure, not code:
- **"Millions of concurrent requests" needs horizontal scaling** — multiple
  container replicas behind a load balancer (Render/ECS/k8s autoscaling),
  not just more workers in one container.
- **Postgres itself has a connection ceiling** (default `max_connections`
  ~100). `num_processes × (DB_POOL_SIZE + DB_MAX_OVERFLOW)` connections
  will exceed that quickly once you scale out — put **PgBouncer** (transaction
  pooling mode) in front of Postgres before you run more than a couple of
  app replicas.
- **JobSpy scraping is bound by the external sites' own rate limits**, not
  by this codebase — no amount of backend tuning makes LinkedIn/Indeed serve
  unlimited scrape traffic. The cache above is what actually absorbs load;
  consider a job-board-side webhook/bulk-export integration if you need
  sustained high query volume.
- **Redis and Postgres should be managed/clustered services at real scale**,
  not the single containers in `docker-compose.yml` (that file is for local
  dev only).


- Skill extraction (`app/utils/skill_extractor.py`) uses a curated keyword
  vocabulary, not an LLM — fast and dependency-free, but will miss skills
  outside the vocabulary. Swap in an LLM-based extractor later without
  changing call sites if higher recall is needed.
- `GET /api/v1/jobs/{job_id}` is not implemented yet (no job persistence
  layer — jobs are fetched live from JobSpy per request, not stored).
